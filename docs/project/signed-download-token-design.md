# MuYunFileServer Signed Download Token Design Draft

本文档记录一个面向后续演进的下载授权方案草案：由业务后端签发短时下载 token，文件服务本地验签后放行下载。

本文档不是一期正式契约，也不代表当前代码已经实现；它用于指导后续设计评审和实现拆解。

---

## 1. 问题背景

当前一期下载模型建立在“可信上游注入身份头”的前提下：

- 浏览器或客户端请求业务网关 / BFF
- 网关 / BFF 完成登录态和业务权限校验
- 网关 / BFF 反向代理到文件服务
- 文件服务基于 `tenant_id`、`user_id`、文件状态完成文件级访问控制后返回文件流

这个模式的优点是简单直接，但对下载链路也有明显代价：

- 大文件流量会持续经过业务网关或 BFF
- 业务网关承担更多长连接、带宽和回源压力
- 前端虽然已授权，但仍需要依赖较重的下载代理路径

---

## 2. 目标

引入一种更轻的下载授权模式，满足以下目标：

- 业务权限仍由业务后端控制
- 文件服务不理解业务对象授权规则
- 大文件下载不必长期穿过业务网关或 BFF
- 前端仍以简单 URL 下载为主，不需要理解文件服务身份头
- 对现有“可信身份头下载”模式保持兼容

---

## 3. 核心思路

推荐采用“短时签名下载 token”模式，而不是一开始就采用严格的一次性随机 token。

典型流程：

1. 用户向业务后端发起“下载某个附件”的请求。
2. 业务后端完成业务权限校验，确认该用户有权下载该附件。
3. 业务后端为该附件签发一个短时有效的下载 token。
4. 业务后端返回一个带 token 的下载 URL，或者直接返回 `302 Location` 跳转到文件服务下载 URL。
5. 文件服务验证 token，通过后直接返回文件流。

推荐下载入口示例：

```text
GET /api/v1/public/files/{fileId}/download?access_token=...
```

这个入口与现有基于身份头的下载入口并存：

- 现有入口：`GET /api/v1/files/{fileId}/download`
- 新入口：`GET /api/v1/public/files/{fileId}/download?access_token=...`

---

## 4. 为什么优先做“短时签名”而不是“严格一次性”

严格一次性 token 看起来更安全，但在浏览器和下载客户端场景里有额外复杂度：

- 浏览器、代理或下载器可能重试请求
- 某些客户端会先做探测请求再真正下载
- 用户点击一次下载，实际可能产生多个 HTTP 请求
- 未来若引入 `Range`，一次性 token 兼容成本会更高

因此更稳妥的第一阶段方案是：

- token 很短时有效，例如 `30` 秒到 `120` 秒
- token 绑定单个 `file_id`
- token 绑定租户
- 可选绑定发起用户
- 文件服务本地验签，不依赖业务后端在线校验

如果后续确有高安全场景，再叠加“一次性消费”能力。

---

## 5. 推荐 token 载荷

建议 token 至少包含以下字段：

| 字段 | 必填 | 说明 |
|---|---|---|
| `iss` | 是 | 签发方标识，例如某个业务系统 |
| `sub` | 否 | 用户标识或主体标识 |
| `tenant_id` | 是 | 当前租户 |
| `file_id` | 是 | 允许下载的文件 ID |
| `exp` | 是 | 过期时间 |
| `iat` | 否 | 签发时间 |
| `jti` | 否 | token 唯一 ID，用于未来扩展一次性消费 |
| `disposition` | 否 | 建议下载模式，默认可省略 |
| `filename` | 否 | 业务方建议文件名，通常可不覆盖服务端原始文件名 |

推荐约束：

- `file_id` 必须与 URL 中的 `{fileId}` 一致
- `tenant_id` 必须与文件元数据中的租户一致
- `exp` 应保持很短
- 若携带 `sub`，仅用于审计和日志，不建议让文件服务回查业务权限

---

## 6. 第一阶段落地范围

为控制实现复杂度，第一阶段只包含以下范围：

- 新增一个基于 token 的下载入口
- 保留现有基于可信身份头的下载入口
- 只支持短时签名 token
- 文件服务本地验签，不回调业务后端
- token 仅用于下载，不扩展到上传、删除、查询
- token 默认不实现严格一次性消费
- token 下载入口仍返回附件下载，不引入 `inline` 和 `Range`

第一阶段明确不做：

- token 撤销列表
- `jti` 消费记录
- 多次下载次数限制
- 按客户端 IP、UA、设备指纹绑定
- 多签发方动态发现
- 预签名上传

---

## 7. 下载接口草案

### 7.1 新增接口

- 方法：`GET`
- 路径：`/api/v1/public/files/{fileId}/download`
- Query 参数：`access_token`

请求示例：

```text
GET /api/v1/public/files/01JABCDEF1234567890ABCDEF/download?access_token=eyJ...
```

### 7.2 行为语义

- 该接口不再要求 `X-Tenant-Id`、`X-User-Id`
- 文件服务完全依赖 `access_token` 做放行判断
- `fileId` 仍保留在路径中，便于日志、路由和下载地址识别
- 成功时返回值与现有下载接口保持一致

### 7.3 成功响应

成功响应应与现有 `GET /api/v1/files/{fileId}/download` 保持一致：

- 状态码：`200 OK`
- 响应体：文件二进制流
- 响应头至少包含：
  - `Content-Type`
  - `Content-Length`
  - `Content-Disposition`

### 7.4 失败场景

建议按以下状态码处理：

| 场景 | 状态码 | 说明 |
|---|---|---|
| 缺少 `access_token` | `401` | 视为未提供有效下载凭证 |
| token 格式非法 | `401` | 不暴露过多解析细节 |
| token 验签失败 | `401` | 统一视为无效凭证 |
| token 已过期 | `401` | 前端或业务后端应重新申请下载链接 |
| URL `fileId` 非法 | `400` | 路径参数本身错误 |
| token 中 `file_id` 与 URL 不一致 | `403` | 凭证与资源不匹配 |
| token 中 `tenant_id` 与文件不匹配 | `403` | 凭证与资源租户不匹配 |
| 文件不存在或已软删 | `404` | 与现有下载语义保持一致 |
| 文件物理内容缺失 | `500` | 属于存储异常 |
| 其他内部错误 | `500` | 未分类内部错误 |

错误返回体建议继续沿用当前统一结构：

```json
{
  "success": false,
  "message": "download token expired",
  "request_id": "01HR...."
}
```

约束：

- `message` 用于展示和排查，不承诺稳定可枚举
- 客户端分支逻辑仍应优先按 HTTP 状态码处理

---

## 8. 业务后端签发接口建议

文件服务只负责验证 token，不负责签发 token。签发入口属于业务后端。

业务后端可选两种返回方式：

### 8.1 返回下载 URL

业务后端接口示例：

```text
POST /biz/files/{fileId}/download-ticket
```

示例响应：

```json
{
  "success": true,
  "data": {
    "download_url": "https://files.example.com/api/v1/public/files/01JABCDEF1234567890ABCDEF/download?access_token=eyJ..."
  }
}
```

### 8.2 直接 302 跳转

业务后端完成权限校验后，直接返回：

- 状态码：`302 Found`
- `Location: https://files.example.com/api/v1/public/files/{fileId}/download?access_token=...`

建议：

- 浏览器下载场景优先使用 `302`
- 若前端需要先拿到 URL 再决定如何下载，可使用 JSON 返回

---

## 9. 签名方式建议

第一阶段建议优先使用“可离线验签”的方案：

### 9.1 HMAC 共享密钥

适合：

- 签发方数量少
- 部署环境简单
- 文件服务与业务后端之间可以安全分发共享密钥

优点：

- 实现最简单
- 文件服务本地即可完成验签

代价：

- 需要妥善管理共享密钥
- 多个业务系统接入时，密钥轮换和隔离会变复杂

### 9.2 非对称签名

适合：

- 未来可能有多个签发方
- 希望由业务后端持有私钥、文件服务只持有公钥

优点：

- 信任边界更清晰
- 便于后续扩展多签发方

代价：

- 实现和配置会略复杂

结论：

- 若后续接入方不多，先用 `HMAC` 就够
- 若从一开始就预期多业务系统接入，可直接采用非对称签名

---

## 10. 配置项建议

文件服务侧建议新增独立配置段，例如：

```yaml
mfs:
  token:
    enabled: false
    issuer: biz-app
    algorithm: hmac-sha256
    secret: change-me
    ttl: 60S
    allowed-clock-skew: 5S
```

建议字段：

| 配置项 | 必填 | 说明 |
|---|---|---|
| `mfs.token.enabled` | 是 | 是否启用 token 模式 |
| `mfs.token.issuer` | 否 | 期望的 `iss`，可用于限制签发方 |
| `mfs.token.algorithm` | 是 | 验签算法，如 `hmac-sha256` |
| `mfs.token.secret` | HMAC 时是 | 共享密钥 |
| `mfs.token.public-key` | 非对称签名时是 | 公钥内容或公钥路径 |
| `mfs.token.allowed-clock-skew` | 否 | 允许的时钟偏差 |

说明：

- `ttl` 更适合作为业务后端签发侧的默认值，不一定由文件服务强制使用
- 文件服务真正强依赖的是验签材料和时间校验参数
- 若 `enabled=false`，则新入口不应暴露

---

## 11. 文件服务校验流程

文件服务收到带 token 的下载请求后，建议按以下顺序校验：

1. 解析 URL 中的 `fileId` 和 `access_token`
2. 校验 token 格式是否合法
3. 验证签名是否合法
4. 验证 `exp` 是否已过期
5. 验证 token 中的 `file_id` 是否与 URL 一致
6. 查询文件元数据
7. 验证文件状态是否为 `ACTIVE`
8. 验证 token 中的 `tenant_id` 是否与文件记录一致
9. 通过后返回文件流

若任一步骤失败，应直接拒绝，不再依赖业务后端在线回查。

---

## 12. 日志与审计建议

token 下载入口应补充独立日志字段，便于和现有下载日志区分。

建议至少记录：

- `operation=download_by_token`
- `result`
- `file_id`
- `tenant_id`
- `request_id`
- `token_issuer`
- `token_subject`
- `reason`

约束：

- 不记录完整 `access_token`
- 不记录完整签名材料
- 若需要定位，可记录脱敏后的 `jti` 或摘要值

---

## 13. 业务后端职责

业务后端在这个模型下仍然是授权决策方，职责包括：

- 校验当前登录用户是否可访问目标业务对象
- 校验该业务对象是否真的关联目标 `file_id`
- 决定是否允许下载
- 决定 token 的过期时间和签发策略
- 将 token 作为下载 URL 或 `302` 跳转返回给前端

文件服务不应承担以下职责：

- 理解业务单据、工单、合同等业务对象关系
- 回查业务数据库判断权限
- 根据业务角色模型做授权决策

一句话：

- 业务后端负责“这个人该不该下载这个附件”
- 文件服务负责“这个 token 是否允许下载这个文件”

---

## 14. 对前端的影响

对浏览器前端来说，这个方案比“所有下载都走业务网关代理”更轻：

- 前端仍先调用业务后端
- 业务后端返回下载 URL 或 `302`
- 前端最终访问的是文件服务下载地址
- 前端无需知道 `X-Tenant-Id`、`X-User-Id`

因此前端心智模型可以保持简单：

- 用户已登录
- 业务后端决定是否允许下载
- 若允许，则给出一个可直接下载的短时 URL

---

## 15. 为什么不推荐“每次下载都回调业务后端校验”

另一个可选方案是：文件服务拿到 token 后，再同步回调业务后端校验。

这个方案可以工作，但不优先推荐，因为它会带来：

- 下载链路额外网络跳数
- 文件服务对业务后端在线可用性更强依赖
- 更高的延迟和故障耦合
- 和“重网关代理”相比收益有限

因此第一阶段更推荐“文件服务本地验签”的方案。

## 16. 测试清单建议

实现时建议至少覆盖以下测试：

### 16.1 单元测试

- token 解析成功
- token 缺失必要字段
- token 验签失败
- token 过期
- `file_id` 不一致
- `tenant_id` 不一致
- 允许的时钟偏差边界

### 16.2 接口级集成测试

- 有效 token 下载成功
- 缺少 token 返回 `401`
- 非法 token 返回 `401`
- 过期 token 返回 `401`
- token 与路径 `fileId` 不匹配返回 `403`
- 文件不存在返回 `404`
- 软删文件返回 `404`
- 物理文件缺失返回 `500`

### 16.3 回归测试

- 现有基于身份头的下载接口行为不变
- `local` 与 `minio` 两种存储模式都能覆盖 token 下载成功路径

## 17. 未来可选增强

若后续安全要求更高，可以在此基础上继续增强：

- `jti` 一次性消费记录
- 更短的过期时间
- 绑定客户端特征，例如 `client_id`
- 绑定下载次数上限
- 支持撤销某个签发方密钥
- 支持多签发方和密钥轮换
- 若未来支持 `Range`，补充分段下载和断点续传下的 token 复用策略

这些都应作为第二阶段或第三阶段增强，不建议在第一版方案里一次做完。

## 18. 开发任务拆解建议

建议按以下顺序推进：

1. 在配置模型中新增 `token` 配置段
2. 实现 token 解析与验签组件
3. 为文件服务新增 token 下载入口，不替换现有身份头入口
4. 打通 token 校验失败和下载成功的错误映射
5. 补充单元测试与接口级集成测试
6. 最后再评估是否需要一次性消费能力

### 18.1 配置与模型

建议新增：

- `FileServiceConfig.DownloadToken`
- 是否启用的布尔开关
- 验签算法配置
- HMAC 密钥或公钥配置
- 允许时钟偏差配置

建议约束：

- 默认关闭
- 启用后必须满足最小配置校验
- 配置缺失应在启动阶段失败，而不是请求时失败

### 18.2 验签与 token 领域组件

建议新增一组独立组件，避免把 token 解析逻辑塞进 resource 或 service：

- `DownloadTokenClaims`
- `DownloadTokenVerifier`
- `DownloadTokenVerificationResult` 或等价模型

建议职责划分：

- `DownloadTokenClaims` 只承载已解析的 token 字段
- `DownloadTokenVerifier` 负责验签、过期校验和 claims 基础校验
- 下载应用服务负责将 claims 与文件元数据做资源级比对

### 18.3 应用服务与资源接口

建议新增：

- `TokenDownloadService` 或在现有下载应用服务上新增 token 下载分支
- `GET /api/v1/public/files/{fileId}/download` 对应的 resource 方法

建议原则：

- 不改写现有基于身份头的下载入口
- token 下载入口与现有下载返回格式保持一致
- token 下载入口不依赖 `RequestContextFilter`

### 18.4 异常映射与错误语义

建议新增或补充：

- `InvalidDownloadTokenException`
- `ExpiredDownloadTokenException`
- `DownloadTokenMismatchException`

建议映射目标：

- 无效 / 缺失 / 过期 token -> `401`
- token 与目标文件不匹配 -> `403`
- 文件不存在或已删 -> `404`
- 存储异常 -> `500`

### 18.5 测试包分组

建议按下面几组补测试：

- token 解析和验签单元测试
- token 下载应用服务单元测试
- `FilesResource` token 下载接口集成测试
- `local` 模式端到端测试
- `minio` 模式端到端测试

### 18.6 文档与接入说明

实现完成后建议同步更新：

- `README.md`
- `docs/design/api.md`
- release 运行说明

重点补充：

- 业务后端如何签发下载 URL
- 浏览器如何使用 `302` 或直接下载 URL
- token 模式与现有可信身份头模式的适用场景

---

## 19. 当前结论

结合当前系统定位，推荐的后续演进方案是：

- 保留现有基于可信身份头的下载入口
- 新增基于短时签名 token 的公开下载入口
- 默认采用本地离线验签，而不是回调业务后端
- 第一阶段优先采用短时签名 token，不做严格一次性消费
- 业务后端继续承担业务授权
- 文件服务只承担 token 验签与文件流下发
- “严格一次性 token”暂不作为第一阶段目标

这个方案能明显降低业务网关的文件流量压力，同时不把业务权限逻辑污染进文件服务。
