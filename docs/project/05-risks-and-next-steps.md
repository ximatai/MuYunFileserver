# MuYunFileServer Risks And Next Steps

本文档记录当前一期实现已经具备的能力边界，以及仍需关注的风险和后续完善事项。

---

## 1. 当前已完成范围

当前代码已实现：

- 多文件上传
- 单文件元数据查询
- 文件下载
- 文件软删
- SQLite migration
- 本地文件系统存储
- 身份头校验与租户隔离
- 软删物理清理定时任务
- 基础健康检查
- 首批单元测试和集成测试

---

## 2. 当前已知风险

### 2.1 上传仍运行在阻塞链路

当前上传服务已标记为 `@Blocking`，这符合一期的 JDBC 和本地文件系统方案，但吞吐能力不会很好。

影响：

- 适合单实例、低到中等并发场景
- 不适合在现阶段宣称高并发上传能力

### 2.2 清理任务仍较为朴素

当前物理清理任务按批次扫描并删除文件与元数据，但没有更细粒度的失败分类、退避策略或专门的补偿记录。

影响：

- 一般失败会依赖下次定时任务重试
- 若出现持续性磁盘异常，需要结合日志排查

### 2.3 MIME 白名单仍是静态配置

当前 MIME 白名单已可配置，但规则粒度仍较粗，没有更细的扩展名组合校验策略和更复杂的安全策略。

影响：

- 一期足够
- 若后续接入更多业务文件类型，需要补充规则设计和测试

### 2.4 下载链路尚未支持更高级 HTTP 能力

当前下载只支持附件下载，不支持：

- `Range`
- `inline`
- 缓存协商
- `ETag`

这符合一期设计，但若文件下载量变大或前端预览需求出现，必须单独演进。

### 2.5 缺少更细粒度的单元测试

目前已有基础单元测试和接口级集成测试，但 repository、异常映射、文件名编码等仍没有覆盖到非常细的粒度。

影响：

- 主流程已有保障
- 底层工具和边界行为仍可继续补强

---

## 3. 建议的下一步

建议按优先级推进：

1. 补更多异常和边界测试，包括下载文件物理缺失、空文件上传、超大文件校验
2. 增加更细的单元测试，特别是 repository 映射和文件名编码
3. 评估是否需要补一个 `/download` 物理缺失场景的集成测试
4. 根据后续业务接入情况，决定是否需要增强 MIME 规则或引入更多审计字段
5. 在准备对接真实业务前，补充部署说明和运行参数示例
