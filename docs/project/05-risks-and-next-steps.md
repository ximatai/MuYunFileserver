# MuYunFileServer Risks And Next Steps

本文档记录当前一期实现已经具备的能力边界，以及仍需关注的风险和后续完善事项。

---

## 1. 当前已完成范围

当前代码已实现：

- 多文件上传
- 单文件元数据查询
- 文件下载
- 文件软删
- SQLite migration
- 本地文件系统存储
- 身份头校验与租户隔离
- 软删物理清理定时任务
- 基础健康检查
- 首批单元测试和集成测试
- 上传链路第二批边界集成测试
- 上传请求解析与单文件预处理的细粒度单元测试
- `UploadService` 第一轮职责拆分

---

## 2. 当前已知风险

### 2.1 上传仍运行在阻塞链路

当前上传服务已标记为 `@Blocking`，这符合一期的 JDBC 和本地文件系统方案，但吞吐能力不会很好。

影响：

- 适合单实例、低到中等并发场景
- 不适合在现阶段宣称高并发上传能力

### 2.2 清理任务仍较为朴素

当前物理清理任务按批次扫描并删除文件与元数据，但没有更细粒度的失败分类、退避策略或专门的补偿记录。

影响：

- 一般失败会依赖下次定时任务重试
- 若出现持续性磁盘异常，需要结合日志排查

### 2.3 MIME 白名单仍是静态配置

当前 MIME 白名单已可配置，但规则粒度仍较粗，没有更细的扩展名组合校验策略和更复杂的安全策略。

影响：

- 一期足够
- 若后续接入更多业务文件类型，需要补充规则设计和测试

### 2.4 下载链路尚未支持更高级 HTTP 能力

当前下载只支持附件下载，不支持：

- `Range`
- `inline`
- 缓存协商
- `ETag`

这符合一期设计，但若文件下载量变大或前端预览需求出现，必须单独演进。

### 2.5 底层映射与规则约束测试仍不够完整

目前已有基础单元测试、上传链路细粒度单元测试和接口级集成测试，但 repository、异常映射、输入约束和错误消息稳定性仍没有覆盖到非常细的粒度。

影响：

- 主流程已有保障
- 底层工具和边界行为仍可继续补强

---

## 3. 建议的下一步

建议按优先级推进：

1. 继续补 repository 映射、异常映射和输入约束的更细粒度测试
2. 继续收口 `remark`、原始文件名和错误 `message` 的契约边界
3. 评估是否将文件名、MIME、hash 等规则进一步从上传预处理逻辑中下沉
4. 根据后续业务接入情况，决定是否需要增强 MIME 规则或引入更多审计字段
5. 在准备对接真实业务前，补充部署说明和运行参数示例

---

## 4. 分类视图

为便于后续持续推进，可将待优化事项按类型理解为以下几组：

### 4.1 测试与契约

- 异常与边界测试补齐
- 接口字段和错误语义进一步收口
- 文件名、备注、大小等输入约束进一步明确

### 4.2 代码结构

- 上传链路拆分
- 工具逻辑下沉
- 更细粒度单元测试入口补足

### 4.3 运维与日志

- 结构化日志字段统一
- 部署说明和运行参数示例完善
- 清理任务失败场景观测增强

### 4.4 安全与规则

- MIME 白名单细化
- 文件名和扩展名策略增强

### 4.5 后续演进

- 下载能力增强
- 签名下载 token 模式
- 数据库迁移预案
- 存储迁移预案

更细的执行项请以 [Development Plan](./04-development-plan.md) 中的 `Next Optimization Backlog` 为准。
