# MuYunFileServer Delete Token Design Draft

本文档记录一个面向后续演进的删除授权方案草案：由业务后端签发短时删除 token，文件服务在 token 验证通过后执行软删。

本文档不是当前正式接口契约，也不代表当前代码已经实现；它用于记录已确认的删除 token 设计决策，并指导后续实现拆解。

---

## 1. 设计目标

删除 token 方案的目标是：

- 减少业务网关或 BFF 对删除链路的持续代理依赖
- 保持业务授权仍由业务后端控制
- 文件服务不理解业务对象授权规则
- 在不引入过多复杂状态管理的前提下，让前端可通过短时凭证完成删除

与查询 / 下载 token 不同，删除是有副作用的操作，因此设计必须更保守。

---

## 2. 已确认的关键决策

以下决策已确认：

1. 删除 token 不做严格一次性消费
2. 删除 token 采用短时有效策略，而不是长期有效
3. 删除 token 必须单独签发，不能复用读取 token
4. 删除 token 必须包含 `purpose=delete`
5. 删除 token 必须绑定 `tenant_id`
6. 删除 token 必须绑定 `file_id`
7. 删除 token 默认有效期按 `30` 秒设计，可配置
8. 删除入口采用 `DELETE`，不用 `GET`
9. 同一个删除 token 被再次调用时，如果文件已删，则按现有删除语义返回 `404`

---

## 3. 为什么删除不用 GET

虽然从技术上可以通过 `GET` 触发删除，但不建议这样设计，原因是：

- `GET` 应保持无副作用语义
- 浏览器预取、代理探测、链接预览都可能触发 `GET`
- 高风险操作不应通过一个可直接访问的 URL 被误触发

因此删除 token 入口明确采用 `DELETE`。

---

## 4. 删除接口草案

### 4.1 新增接口

- 方法：`DELETE`
- 路径：`/api/v1/public/files/{fileId}`
- Query 参数：`access_token`

请求示例：

```text
DELETE /api/v1/public/files/01JABCDEF1234567890ABCDEF?access_token=eyJ...
```

### 4.2 行为语义

- 该入口不要求 `X-Tenant-Id`、`X-User-Id`
- 文件服务依赖 `access_token` 做删除放行判断
- 业务后端负责业务授权与删除 token 签发
- 文件服务只校验 token 是否允许删除目标文件
- 删除行为与现有 `DELETE /api/v1/files/{fileId}` 保持一致，即执行软删

### 4.3 成功响应

建议成功响应与现有删除接口保持一致：

- 状态码：`200 OK`

```json
{
  "success": true,
  "data": {
    "id": "01JABCDEF1234567890ABCDEF",
    "status": "DELETED",
    "deleted_at": "2026-04-02T10:30:00Z"
  }
}
```

### 4.4 失败场景

建议按以下状态码处理：

| 场景 | 状态码 | 说明 |
|---|---|---|
| 缺少 `access_token` | `401` | 视为未提供有效删除凭证 |
| token 格式非法 | `401` | 不暴露过多解析细节 |
| token 验签失败 | `401` | 统一视为无效凭证 |
| token 已过期 | `401` | 应重新申请删除授权 |
| URL `fileId` 非法 | `400` | 路径参数本身错误 |
| token 中 `purpose` 不是 `delete` | `403` | 读取 token 不能用于删除 |
| token 中 `file_id` 与 URL 不一致 | `403` | 凭证与资源不匹配 |
| token 中 `tenant_id` 与文件不匹配 | `403` | 凭证与资源租户不匹配 |
| 文件不存在或已删除 | `404` | 与现有删除语义保持一致 |
| 内部错误 | `500` | 未分类内部错误 |

说明：

- 同一个 token 若在有效期内被再次调用，而文件已删，应继续返回 `404`
- 当前设计不引入“token 已消费”这一新错误语义

---

## 5. 推荐 token 载荷

删除 token 建议至少包含以下字段：

| 字段 | 必填 | 说明 |
|---|---|---|
| `iss` | 是 | 签发方标识 |
| `sub` | 否 | 用户标识或主体标识 |
| `tenant_id` | 是 | 当前租户 |
| `file_id` | 是 | 允许删除的文件 ID |
| `purpose` | 是 | 固定为 `delete` |
| `exp` | 是 | 过期时间 |
| `iat` | 否 | 签发时间 |
| `jti` | 否 | token 唯一 ID，当前阶段仅用于日志和未来扩展 |

约束：

- `purpose` 必须固定为 `delete`
- `file_id` 必须与 URL 中的 `{fileId}` 一致
- `tenant_id` 必须与文件元数据中的租户一致
- `exp` 应保持很短

---

## 6. 为什么不做严格一次性消费

删除 token 若做严格一次性，需要额外引入：

- token 消费记录表
- 并发抢占与唯一约束
- 消费时机设计
- 删除失败与重试语义
- 更多并发和异常测试

考虑当前项目目标是保持实现轻量，因此已确认第一版不做严格一次性消费，而采用：

- 短时有效
- 强绑定 `purpose`
- 强绑定 `tenant_id`
- 强绑定 `file_id`

这意味着删除 token 是“短时间内可重复请求的删除授权”，不是“严格单次授权”。

在当前软删模型下，其直接后果通常是：

- 第一次删除成功返回 `200`
- 后续在有效期内重复调用时，由于文件已删，返回 `404`

---

## 7. 业务后端职责

业务后端仍然是删除授权的决策方，职责包括：

- 校验当前登录用户是否有权删除目标业务对象关联的附件
- 确认该业务对象真的关联目标 `file_id`
- 决定是否允许删除
- 签发 `purpose=delete` 的短时 token
- 将删除 URL 或 token 返回给前端

文件服务不应承担以下职责：

- 理解业务对象关系
- 理解业务角色权限
- 回查业务数据库判断删除权限

一句话：

- 业务后端负责“这个人该不该删这个附件”
- 文件服务负责“这个 token 是否允许删这个文件”

---

## 8. 前端交互模式

删除 token 不建议用跳转或 `302` 语义触发，而应由前端显式发起 `DELETE` 请求。

推荐流程：

1. 前端发起“删除附件”请求到业务后端
2. 业务后端完成业务权限校验
3. 业务后端返回删除 URL 或短时 token
4. 前端在用户确认后，显式调用：

```text
DELETE /api/v1/public/files/{fileId}?access_token=...
```

这样比跳转更清晰，也更符合副作用操作的心智模型。

---

## 9. 配置项建议

文件服务可沿用与读取 token 相同的验签配置模型，但建议补充独立开关，例如：

```yaml
mfs:
  delete-token:
    enabled: false
    ttl: 30S
```

若不想单独拆配置段，也可以复用既有 token 验签材料，仅新增删除能力开关。

建议最少支持：

| 配置项 | 必填 | 说明 |
|---|---|---|
| `mfs.delete-token.enabled` | 是 | 是否启用删除 token 入口 |
| `mfs.delete-token.ttl` | 否 | 建议默认有效期 |

说明：

- `ttl` 更适合作为业务后端签发默认值
- 文件服务更关心验签材料与过期校验

---

## 10. 文件服务校验流程

文件服务收到带删除 token 的请求后，建议按以下顺序校验：

1. 解析 URL 中的 `fileId` 和 `access_token`
2. 校验 token 格式是否合法
3. 验证签名是否合法
4. 验证 `exp` 是否已过期
5. 验证 `purpose` 是否为 `delete`
6. 验证 token 中的 `file_id` 是否与 URL 一致
7. 查询文件元数据
8. 验证文件状态是否为 `ACTIVE`
9. 验证 token 中的 `tenant_id` 是否与文件记录一致
10. 通过后执行软删

---

## 11. 日志与审计建议

建议至少记录以下字段：

- `operation=delete_by_token`
- `result`
- `file_id`
- `tenant_id`
- `request_id`
- `token_issuer`
- `token_subject`
- `reason`

约束：

- 不记录完整 `access_token`
- 不记录完整签名材料
- 若需要定位，可记录脱敏后的 `jti` 或摘要值

---

## 12. 测试清单建议

实现时建议至少覆盖以下测试：

### 12.1 单元测试

- `purpose=delete` 校验成功
- 读取 token 误用于删除时返回拒绝
- token 过期
- token 验签失败
- `tenant_id` 不匹配
- `file_id` 不匹配

### 12.2 接口级集成测试

- 有效删除 token 删除成功
- 缺少 token 返回 `401`
- 非法 token 返回 `401`
- 过期 token 返回 `401`
- `purpose` 不正确返回 `403`
- 文件不存在返回 `404`
- 已删除文件返回 `404`

### 12.3 回归测试

- 现有 `DELETE /api/v1/files/{fileId}` 头模式行为不变
- 读取 token 模式行为不变

---

## 13. 当前结论

结合当前项目目标，推荐的删除 token 方案是：

- 新增 `DELETE /api/v1/public/files/{fileId}?access_token=...`
- 删除 token 必须单独签发
- 删除 token 必须带 `purpose=delete`
- 删除 token 绑定 `tenant_id` 和 `file_id`
- 默认有效期按 `30` 秒设计
- 第一版不做严格一次性消费
- 重复删除时仍按现有语义返回 `404`

这个方案在复杂度和安全性之间取了一个偏务实的平衡点，适合当前项目阶段。
