# MuYunFileServer Overview Design

## 1. 文档目标

本文档定义 `MuYunFileServer` 一期的正式设计边界、核心约束和实现原则，作为后续任务拆解、接口设计和实现开发的上位依据。

本文档只描述一期设计，不将二期及以后能力混入一期实现目标。

---

## 2. 一期定位

`MuYunFileServer` 一期定位为一个独立的文件资产服务，目标是提供稳定、轻量、可演进的文件存取能力，而不是网盘系统、资产管理平台或预览平台。

一期明确聚焦以下能力：

- 文件上传
- 文件下载
- 文件删除
- 单文件元数据查询
- 基础文件访问控制
- 元数据持久化

一期明确不实现以下能力：

- 文件预览
- OCR
- 文档转码
- 视频转码
- 病毒扫描
- 对象存储直传
- 分片上传
- 断点续传
- HTTP Range
- 公开分享链接
- 分页列表和复杂搜索
- 文件资产治理
- 管理端接口

---

## 3. 总体设计原则

- 一期以轻量、清晰、可落地为优先，不为未来能力预埋过重结构。
- 文件服务只承担文件领域职责，不承担业务对象治理职责。
- 一期优先保证上传、下载、删除和元数据链路稳定。
- 所有未来扩展能力都应通过抽象层和平滑迁移演进，不反向污染一期设计。

---

## 4. 技术路线

### 4.1 框架选型

- 一期采用 `Quarkus + Vert.x`
- 使用异步友好的运行时能力处理文件上传、下载和流式 IO
- 一期不引入独立消息队列或通用异步任务系统

### 4.2 部署形态

- 一期按单实例部署设计
- 一期不解决多实例共享存储和分布式一致性问题
- 未来是否演进为多实例部署，由实际规模和可用性需求决定

### 4.3 存储选型

- 一期使用本地文件系统存储文件二进制内容
- 系统内部通过 `StorageProvider` 抽象隔离存储实现
- 对外模型、数据库模型和接口协议均不得暴露本地文件路径语义
- 未来允许替换为 `MinIO/S3`

### 4.4 数据库选型

- 一期使用 `SQLite` 作为元数据数据库
- 数据访问采用显式 SQL 为主，不使用 ORM
- Schema 变更使用版本化 SQL migration 管理
- 应用启动时执行 migration
- SQL 与表结构设计应保持未来迁移 `PostgreSQL` 的可能性

---

## 5. 租户、身份与信任边界

### 5.1 多租户策略

- 整体按多租户模型设计
- 文件记录必须包含 `tenant_id`
- 一期不支持跨租户共享文件
- 一期不自行推导租户归属

### 5.2 身份上下文

- 上游至少透传 `tenant_id` 和 `user_id`
- 可选透传 `request_id`、`client_id`、用户名等增强字段
- `tenant_id` 和 `user_id` 均不作为 URL 路径参数

### 5.3 信任边界

- 文件服务一期不自行实现认证能力
- 所有请求必须经由统一网关或受控上游进入
- 文件服务统一信任由网关或受控上游注入的身份上下文
- 文件服务不接受客户端自行伪造的身份 header
- 网关负责认证、身份解析、租户解析和身份注入
- 业务系统负责业务语义授权
- 文件服务负责文件级访问控制、租户隔离、状态校验和审计

### 5.4 前端访问模式

- 一期允许前端通过统一网关或受控入口直接访问文件服务
- 业务系统不承担文件流量中转职责

---

## 6. 文件模型与生命周期

### 6.1 文件模型

- 一期采用不可变文件模型
- 文件上传后内容不可原地覆盖
- 如需替换文件，由业务系统创建新的文件记录并更新自己的引用关系

### 6.2 删除模型

- 一期采用软删语义
- 删除操作仅更新文件状态并记录 `deleted_at`
- 文件自软删起立即不可访问
- 对已软删文件的普通查询和下载请求统一按 `404 Not Found` 处理
- 一期不提供面向业务方的恢复接口
- 一期不对业务方暴露物理清理接口

### 6.3 内部物理清理

- 系统通过内部定时任务对已软删文件执行物理清理
- 默认保留期为 7 天，且该值可配置
- 清理条件为：文件已软删且超过保留期
- 物理清理失败时允许后续重试

### 6.4 文件状态

一期最小文件状态集为：

- `ACTIVE`
- `DELETED`

一期不引入预览、审核、转码等额外状态。

---

## 7. 文件与业务对象关系

- 文件服务一期不维护业务对象关联关系
- 业务系统自行维护 `file_id` 与业务对象的关系
- 文件服务只负责文件本体与元数据
- 一期不承担孤儿文件识别、引用关系盘点、资产检索或存在性批量校验等治理职责

---

## 8. 文件标识与存储命名

### 8.1 file_id 规则

- 文件对外标识使用 `ULID`
- `file_id` 与底层 `storage_key` 解耦
- 不使用数据库自增主键作为对外标识

### 8.2 请求端显式指定 file_id

- 一期允许请求端可选指定 `file_id`
- 若请求端提供 `file_id`，则必须是合法 `ULID`
- 服务端负责 `file_id` 的格式校验与唯一性校验
- 若 `file_id` 已存在，则返回 `409 Conflict`
- 不允许覆盖、复用或借软删状态重新占用已有 `file_id`
- 若未提供 `file_id`，则由服务端生成 `ULID`

### 8.3 storage_key 规则

- `storage_key` 采用独立对象键语义，不使用原始文件名作为物理文件名
- `storage_key` 应包含租户维度和日期分片信息
- 本地文件系统目录结构应按可迁移方式组织，避免散乱路径设计

建议结构示例：

```text
tenant/{tenantId}/yyyy/MM/dd/{ulid}
```

---

## 9. 元数据模型

### 9.1 核心字段

一期核心元数据字段包括：

- `id`
- `tenant_id`
- `original_filename`
- `extension`
- `mime_type`
- `size_bytes`
- `sha256`
- `storage_provider`
- `storage_key`
- `storage_bucket` 或等价字段
- `status`
- `uploaded_by`
- `uploaded_at`
- `deleted_at`
- `delete_marked_by`
- `remark`

### 9.2 模型边界

- 一期元数据以通用文件资产字段为主
- 不放业务耦合字段进入核心表
- 不支持业务标签体系，仅保留 `remark` 作为轻量备注字段
- 原始文件名允许重复，仅作为展示属性，不参与唯一性约束
- 服务端保留最终 MIME type 判定权，并将最终值写入元数据
- `storage_provider`、`storage_key`、本地路径等存储实现字段不作为对外接口正式契约
- 一期不预留预览相关字段
- 一期不提取和保存内容摘要信息，如图片尺寸、PDF 页数、视频时长等

---

## 10. 上传设计

### 10.1 上传模式

- 一期采用服务端中转上传
- 客户端通过 `multipart/form-data` 或等价 HTTP 方式上传到文件服务
- 一期不实现上传会话或两阶段上传
- 一期不支持分片上传

### 10.2 上传大小与处理方式

- 单文件大小上限为 `500MB`
- 上传处理必须采用流式落盘和流式 hash 计算
- 不允许将整个文件一次性加载到内存
- 上传成功后文件立即可查询和下载

### 10.3 多文件上传

- 一期支持单次请求上传多个文件
- 单次请求最多上传 10 个文件
- 多文件上传采用整单成功 / 整单失败语义
- 任一文件校验失败或写入失败，整次上传失败
- 服务端应回滚本次请求中已写入的正式文件与元数据记录

### 10.4 上传资源保护

为控制本地文件系统与 SQLite 组合下的资源风险，一期对上传链路施加以下约束：

- 服务端在接收上传前执行磁盘剩余空间阈值校验
- 磁盘空间不足时拒绝上传
- 上传过程使用独立临时目录
- 启动时和定时任务都需要清理残留临时文件
- 若文件已落盘但元数据写库失败，服务端应立即尝试删除已落盘文件进行回滚
- 若回滚删除失败，则记录结构化错误日志并纳入后续内部清理任务

### 10.5 上传原子性顺序

推荐实现顺序如下：

1. 校验请求与身份上下文
2. 校验上传数量、单文件大小、磁盘阈值
3. 将文件流写入临时目录并计算 `sha256`
4. 生成或校验 `file_id`
5. 将临时文件移动到正式存储位置
6. 写入元数据记录
7. 返回成功结果

若任一关键步骤失败，应执行整次上传失败回滚与日志记录。

---

## 11. 下载设计

- 一期采用文件服务代理下载
- 文件服务完成身份校验、文件状态校验和租户校验后输出文件流
- 一期下载接口仅提供附件下载语义
- 默认 `Content-Disposition: attachment`
- 下载响应中保留原始文件名作为建议下载名
- 一期不提供内联展示模式切换
- 一期不支持 HTTP Range 或断点续传下载

---

## 12. 删除设计

- 删除接口仅执行软删
- 删除请求应校验租户边界与文件状态
- 删除成功后文件立即不可访问
- 软删后的物理清理由内部定时任务处理
- 一期不对业务方提供 purge 或恢复接口

---

## 13. 查询与接口边界

### 13.1 查询能力

- 一期仅提供单文件维度元数据查询能力
- 不提供分页列表、复杂搜索、全文检索或目录视图
- 不单独提供存在性校验接口
- 文件是否存在及是否可访问，通过单文件元数据查询接口体现

### 13.2 批量能力

- 一期仅支持批量上传
- 不提供批量删除、批量元数据查询和批量下载

### 13.3 接口版本

- 对外接口从一期开始采用版本化路径
- 默认使用 `/api/v1` 作为前缀

### 13.4 返回与错误契约

- 一期不定义独立业务错误码体系
- 优先使用标准 HTTP 状态码表达错误
- 无法细分建模的内部异常统一按 `500` 处理
- 即使不使用业务错误码，接口也应保持稳定的错误返回体结构
- 错误响应至少应包含可读错误信息与请求关联信息

---

## 14. 安全与校验

- 一期支持基于配置的基础文件类型安全限制
- 主要依据服务端识别的 MIME type，并结合原始文件名扩展名辅助判断
- 默认拦截明显高风险类型
- 规则必须配置化
- 一期不做复杂内容安全扫描
- 一期不做病毒扫描

---

## 15. 可观测性与运维基线

### 15.1 健康检查

- 一期提供基础健康检查能力
- 至少包括 `liveness` 和 `readiness`

### 15.2 日志与审计

- 一期需要轻量审计能力
- 不单独建设审计表
- 关键操作通过结构化日志记录，由 `logback` 输出

至少记录以下操作：

- 上传成功
- 上传失败
- 下载成功
- 下载失败
- 删除成功
- 删除失败
- 单文件元数据查询

建议日志至少包含以下字段：

- 时间
- 操作类型
- file_id
- tenant_id
- user_id
- request_id
- 请求来源
- 结果
- 失败原因

### 15.3 统计策略

- 一期不在主库中维护下载次数或访问次数统计
- 如需统计分析，后续从日志侧聚合

### 15.4 配置管理

- 一期采用配置文件与环境变量结合的方式管理配置
- 不引入独立配置中心
- 关键运行参数必须可配置，不得硬编码在业务代码中

建议至少配置化以下项目：

- 文件根目录
- 临时目录
- 单文件大小上限
- 单请求最大文件数
- 磁盘剩余空间阈值
- SQLite 路径
- MIME 限制规则
- 软删保留期
- 日志级别
- 服务监听端口

---

## 16. 一期不承担的职责

为避免范围膨胀，一期明确不承担以下职责：

- 文件资产治理
- 业务对象引用治理
- 孤儿文件识别与自动治理
- 资产搜索和盘点
- 对外管理能力
- 预览及预处理能力
- 安全扫描中心能力

一期目标仅限于：受控文件存取与元数据管理。

---

## 17. 后续演进方向

在不改变一期核心边界的前提下，后续可以逐步演进：

- 本地文件系统迁移到 `MinIO/S3`
- `SQLite` 迁移到 `PostgreSQL`
- 引入预览能力
- 引入异步任务与预处理任务框架
- 增加临时链接下载能力
- 增加对象存储直传能力

这些能力均应作为后续阶段独立设计，不应反向影响一期的轻量边界。
